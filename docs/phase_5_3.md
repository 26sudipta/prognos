## Concepts Explained

### 1. Why a PWA at all

Apple does not allow sideloading: shipping a native iOS app requires the $99/year
Apple Developer Program, a Mac for signing, and App Store review (which would also
force adding Sign in with Apple, per guideline 4.8). A PWA sidesteps all of it:
iPhone users open the site in Safari → Share → **Add to Home Screen** and get a
full-screen, app-like PROGNOS for $0. The strategy is therefore:

| Platform | Delivery |
|---|---|
| Android | Native Flutter APK (premium: offline alarms, widget) — unchanged |
| iPhone | **PWA** — everything except reminders |
| Native iOS via App Store | Deferred until the startup phase justifies $99/yr |

### 2. `app/manifest.ts` instead of a static `manifest.json`

Next.js App Router treats `manifest.ts` as a metadata route: it serves
`/manifest.webmanifest` **and injects the `<link rel="manifest">` tag
automatically** — no manual `<head>` wiring, and it's type-checked
(`MetadataRoute.Manifest`). Key choices:

- `start_url: "/dashboard"` — an installed app should open like an app, not a
  marketing page. Unauthenticated users get bounced to `/login` by the existing
  dashboard layout guard, so this is safe.
- `display: "standalone"` — no browser chrome.
- `background_color` / `theme_color: #070B14` — matches `--bg-base`, so the
  splash and status bar blend with the app's dark background.

### 3. Icons: three variants, not one

| File | Why it exists |
|---|---|
| `icon-192/512.png` (rounded) | Standard manifest icons; match the browser-tab favicon (`app/icon.tsx`: indigo `#6366F1`, white trend-line) |
| `icon-maskable-512.png` (full-bleed) | Android "adaptive icons" crop into circles/squircles; a maskable icon keeps the glyph inside the 48%-radius safe zone so nothing gets clipped |
| `app/apple-icon.png` (full-bleed, opaque) | iOS ignores manifest icons for the home screen and uses `apple-touch-icon`; Next.js auto-links any `app/apple-icon.png`. iOS applies its own corner mask, so the file must be square and opaque |

All generated by one Pillow script (16× supersampled, LANCZOS-downsampled) so
the home-screen icon is pixel-identical in style to the favicon.

### 4. The service worker deliberately does almost nothing

`public/sw.js` has exactly three behaviors:

1. **`/api/` is untouched** — the fetch handler returns early, so auth token
   refresh, sync triggering, and leaderboard reads always hit the network.
   Caching API responses would violate the "server is the source of truth"
   architecture and could serve stale/forged-looking leaderboards.
2. **Navigations are network-first** with a branded offline fallback. The
   fallback HTML is **inlined in the worker** (inline CSS, inline SVG) rather
   than a precached Next.js route — a cached route's HTML would still reference
   hashed JS/CSS chunks that may not be cached, rendering broken. The inline
   page always works.
3. **`/_next/static/` and `/icons/` are cache-first** — those URLs contain
   content hashes and are immutable, so caching them is free performance with
   zero staleness risk.

`skipWaiting()` + `clients.claim()` make new worker versions take over
immediately on deploy; bumping the `CACHE` constant invalidates old assets.

### 5. Registration is production-only

`pwa-register.tsx` registers `/sw.js` only when `NODE_ENV === "production"`.
A service worker in dev caches hot-reload assets and causes phantom staleness —
the classic PWA development trap.

### 6. iOS install hint

iOS Safari never prompts users to install a PWA (Android Chrome does). The
`IosInstallHint` banner fills that gap on the landing page:

- Detects iOS via user agent, **including iPadOS 13+** which masquerades as
  macOS (distinguished by `maxTouchPoints > 1`).
- Hides when already installed (`display-mode: standalone` media query or the
  legacy `navigator.standalone`).
- Appears after a 3 s delay (don't ambush visitors), dismissible once via
  `localStorage`.

### 7. What the PWA cannot do

No on-device scheduled notifications — browsers do not expose alarm APIs, so
**contest reminders remain Android-app-exclusive**. The future path is Web Push
(iOS 16.4+ supports it for installed PWAs), which requires a
push-subscriptions table + a cron-triggered send worker on the backend.
Deliberately deferred.

## Verification

```bash
cd frontend && npm run build          # 0 TS errors, 0 ESLint errors;
                                      # route list shows /manifest.webmanifest + /apple-icon.png

npm start -- -p 3457                  # then:
curl -s localhost:3457/manifest.webmanifest   # full manifest JSON, start_url /dashboard
curl -s localhost:3457/ | grep -oE '<link rel="manifest"[^>]*>|<meta name="theme-color[^>]*>'
                                      # → both tags present in <head>
curl -s -o /dev/null -w "%{http_code}" localhost:3457/sw.js            # 200
curl -s -o /dev/null -w "%{http_code}" localhost:3457/icons/icon-512.png  # 200
```

On-device (after deploy): Chrome DevTools → Application → Manifest shows
"Installable"; iPhone Safari → Share → Add to Home Screen → PROGNOS opens
full-screen with the indigo icon.

## Key Takeaways

- A PWA is the only $0 route to iPhone users; the native iOS app is a business
  decision ($99/yr + Sign in with Apple), not a technical one.
- Keep service workers minimal and **never cache the API** — installability and
  static-asset caching deliver 95% of the value with none of the staleness bugs.
- iOS needs its own icon (`apple-icon.png`) and its own install UX (manual
  Add-to-Home-Screen hint) — the manifest alone only covers Android/desktop.
- Inline the offline fallback in the worker; precached framework routes break
  when their hashed chunks aren't cached.
