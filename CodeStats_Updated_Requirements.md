# CodeStats: Unified Competitive Programming Analytics & Planning Platform
## Updated Product Requirements Document (PRD)

### 1. Executive Summary
CodeStats is an API-first analytics and planning platform designed to aggregate competitive programming contest schedules and user activity data into a unified dashboard. Originally conceptualized as a strictly web-first, single-user system, the platform's vision has evolved to support a multi-tenant B2B **Classroom** model and a mobile companion app. The core objective remains: to transform raw, fragmented practice activity into actionable insights (heatmaps, streaks, topic-wise analytics) and foster consistent improvement through structured guidance and gamified peer competition.

### 2. Rollout Strategy
* **Phase 1: Web-First Core (V1):** The initial launch will focus entirely on the web application and backend API. This reduces development overhead, provides the optimal environment for rich analytics dashboards, and establishes the "Hub" for Classroom management.
* **Phase 5: Mobile Companion App:** Once the web platform is stable, development will shift to a streamlined mobile app focused on edge engagement (local device alarms and quick dashboards).

### 3. Core Feature Modules

#### Module A: Authentication & Multi-Tenancy (The Classroom Framework)
Introducing Classrooms changes the core access pattern from a single-user system to a role-based, multi-tenant ecosystem.
* **Roles:** * `Standard User / Student`
    * `Teacher / Mentor`
* **Handle Verification Protocol:** Users must link external accounts (starting with Codeforces). To prevent cheating, verification requires the user to temporarily change their Codeforces profile summary to a randomized string generated by CodeStats during registration.
* **Classroom Lifecycle:**
    * Teachers can create classrooms and generate secure, time-limited invite links.
    * Students can join via the invite token and can belong to multiple classrooms (e.g., University Course, CP Club).
* **Full Transparency Leaderboard:** To foster a "Strava for Competitive Programming" competitive mindset, classroom members can view the comprehensive stats of their peers (activity heatmaps, solved counts, tag weaknesses). Students must grant this visibility upon joining.

#### Module B: Core Analytics Engine (Rule-Based)
* **Data Ingestion:** A background worker incrementally fetches data from the Codeforces API (submissions, ratings, tags) and CLIST (contests).
* **Unified Dashboard:** Displays a GitHub-style activity heatmap, current streak, longest streak, and rating trend lines.
* **Skill Matrix:** Calculates success ratios per problem tag (e.g., DP, greedy, math). Records failure types (Wrong Answer, TLE, etc.) to build an error profile.
* **Weakness Detection & Recommendations:** Rule-based flagging of under-practiced or high-failure-rate tags to suggest specific difficulty brackets for future practice. *(Note: AI/LLM integration is deferred to a future phase).*

#### Module C: Mobile Edge & Contest Discovery
To provide a competitive edge without inflating server costs, the mobile platform must execute processing natively.
* **Offline Synchronization:** The mobile application caches the upcoming contest feed locally via an embedded database.
* **Zero-Cost Notifications (Native Device Alarms):** Instead of using expensive Firebase Cloud Messaging (FCM) or server cron jobs, the app uses OS-level background fetch tasks. It silently pulls lightweight schedule updates and relies entirely on the device's native hardware clock to schedule local alarms.

### 4. Technology Stack

**Client Layer (Frontends)**
* **Web Frontend:** Next.js (TypeScript), Tailwind CSS + shadcn/ui, Recharts/Chart.js, date-fns/Luxon.
* **Mobile App:** Flutter (with `flutter_local_notifications` and `workmanager` for background tasks).
* *Note:* Frontends are "dumb"—they only read pre-calculated data from the database.

**API Layer (Backend)**
* **Core Framework:** FastAPI (Python) for fast, async request handling.
* **Data Handling & Security:** Pydantic (validation), SQLAlchemy & Alembic (ORM), JWT Authentication.

**Data & Processing Layer (The Engine Room)**
* **Primary Database:** PostgreSQL.
* **Background Worker:** Celery + Redis. Essential for incrementally syncing heavily rate-limited external APIs (Codeforces) and computing heavy aggregations.
